# Технологии Starmap

## Основной стек технологий

### Backend/Frontend Framework
- **Ruby on Rails 8.1.1** - Основной фреймворк приложения
- **Hotwire (Turbo + Stimulus)** - Для интерактивного UI без отдельного frontend фреймворка
  - **Turbo**: Ускоряет навигацию и обновления страниц
  - **Stimulus**: Контроллеры для интерактивности

### База данных
- **PostgreSQL 1.1** - Основная база данных
- **pg gem** - Адаптер для PostgreSQL

### Веб-сервер
- **Puma** - Concurrent web server (версия >= 5.0)

### Аутентификация и авторизация
- **Devise** - Основная система аутентификации с базой данных
- **Pundit** - Система авторизации на основе политик

### Authentication
- **Devise**: базовая аутентификация с базой данных
- **Session management**: Rails session + Devise

### Authorization
- **Pundit**: декларативная авторизация
- **Role-based access**: 4-уровневая система ролей
- **Record-level**: проверка принадлежности записей

#### Стандартные принципы Pundit
- **Policy Scope**: `policy_scope(Model)` для фильтрации коллекций по правам доступа
- **Авторизация объектов**: `authorize @object` для проверки конкретных записей
- **Символьная авторизация**: `authorize :model, :action?` для действий без объектов

#### Исключения из стандартной схемы
Иногда возникает необходимость обойти стандартную схему Pundit:

**Когда использовать:**
- **Контроллеры без коллекций**: Дашборды, агрегированные отчеты
- **Специальные действия**: Методы, не связанные с CRUD операциями
- **Кастомная логика**: Когда стандартный scoping не подходит

**Как обойти:**
```ruby
# В контроллере
skip_after_action :verify_policy_scoped  # Для отключения проверки scope
skip_after_action :verify_authorized     # Для отключения проверки authorize

# В политике
authorize :dashboard, :overview?         # Авторизация через символы
authorize current_user, :personal?       # Авторизация конкретного пользователя
```

## Архитектурные решения

### Выбор технологий
1. **Rails 8.1.1**: Последняя версия с улучшенной производительностью
2. **Hotwire**: Современный подход к интерактивности без JavaScript фреймворков
3. **Solid Queue**: Встроенная система очередей Rails, замена внешних решений
4. **PostgreSQL**: Надежная реляционная база с расширенными возможностями

### Безопасность
- CSRF protection включен по умолчанию
- Фильтрация чувствительных параметров в логах
- Pundit для декларативной авторизации
- Devise с базой данных для аутентификации

### Производительность
- Eager loading для предотвращения N+1 запросов
- Кеширование метрик через Solid Cache
- Индексы на критических полях
- Оптимизированные scopes в моделях

### Развертывание
- **Docker/Docker Compose** - Контейнеризация приложения
- **Multi-stage build** - Оптимизация размера Docker образов
- Поддержка health checks для мониторинга

## Разработка

### Команды для разработки
```bash
# Запуск сервера разработки
bin/dev

# Консоль Rails
bin/rails console

# Выполнение миграций
bin/rails db:migrate

# Запуск тестов
bin/rails test

# Запуск конркетного тестового файла
bundle exec rspec path/to/some_spec.rb
```

### Оптимизация тестирования
#### Ключевые техники оптимизации
- **let_it_be**: Хелпер из RSpec для создания общих тестовых данных один раз на весь набор тестов
- **build vs create**: Использование `build` вместо `create` для проверки авторизации без сохранения в БД
- **Профилирование тестов**: Инструмент `test-prof` для анализа производительности тестов

#### Принципы оптимизации тестов политик
1. **Общие данные**: Все пользователи, команды и технологии создаются через `let_it_be`
2. **Локальные данные**: Специфичные для теста объекты создаются через `build`
3. **База данных**: Минимизация записей в БД для повышения скорости тестирования
4. **Изоляция**: Каждый тест остается независимым при общих данных
