# ТЕХНИЧЕСКОЕ ЗАДАНИЕ НА РАЗРАБОТКУ

## Продукт: Starmap Web-приложение для управления компетенциями и рисками

### **1. Обзор Продукта и Философия**

Starmap — корпоративное web-приложение, предназначенное для автоматизации процессов сбора, анализа и управления техническими компетенциями команды. Цель — трансформировать управление скрытыми знаниями в проактивную и стратегическую деятельность, обеспечивая прозрачность, устойчивость и целенаправленное развитие.

**Основные принципы:**
*   **Прозрачность:** Создание единой, объективной картины компетенций для принятия взвешенных решений.
*   **Устойчивость:** Формирование команд, где экспертиза эффективно распределена, хеджируя риски и обеспечивая стабильность продуктов.
*   **Осознанное Развитие:** Превращение профессионального роста из случайного процесса в управляемый и мотивирующий для каждого инженера.

### **2. Целевая Аудитория и Её Мотивация**

*   **Разработчик:** Получает персональный навигатор для карьерного роста, визуализацию прогресса, аргументы для обсуждения грейда и новые возможности для развития.
*   **Тимлид:** Инструмент для построения сильных, сбалансированных команд; объективная основа для 1-on-1, менторства и планирования обучения.
*   **Unit Lead / CTO:** Гарантия стабильности юнита, снижение рисков, обоснование инвестиций в обучение и стратегическое планирование развития команды.

### **3. Технологический Стек**

*   **Backend / Frontend Framework:** Ruby on Rails 8 (с использованием Hotwire: Turbo, Stimulus)
*   **База данных:** PostgreSQL
*   **Очередь фоновых задач:** Rails Solid Queue (вместо Redis/Sidekiq)
*   **Авторизация / Авторизация:** `Devise` для аутентификации, `Action Policy` для авторизации + `Devise LDAP Authenticatable` для синхронизации пользователей с LDAP.
*   **Версионирование/Аудит:** `Audited` (или схожее решение) для отслеживания изменений оценок и планов.
*   **Контейнеризация:** Docker / Docker Compose

### **5. Функциональные Требования**

#### **5.1. Управление Пользователями и Доступом (на основе LDAP)**
*   **F.5.1.1. Синхронизация Пользователей:** Автоматическая синхронизация списка пользователей и групп из LDAP в систему Starmap. Должна выполняться фоновым процессом (Solid Queue) по расписанию (например, ежедневно).
*   **F.5.1.2. Аутентификация:** Все пользователи должны аутентифицироваться через LDAP (SSO).
*   **F.5.1.3. Ролевая Модель:** Внедрение ролевой системы с использованием `Action Policy`. Система должна поддерживать следующие роли:
    *   **Разработчик (Engineer):** Видит свой персональный дашборд и агрегированные данные по всему юниту (`Overview`) без детализации по другим сотрудникам. Может редактировать только собственную самооценку.
    *   **Тимлид (Team Lead):** Имеет доступ ко всей информации своей команды (оценки всех сотрудников, все дашборды, планы действий). Может **утверждать или корректировать** оценки своей команды.
    *   **Unit Lead:** Имеет доступ ко всем данным и функционалу любых команд в юните.
    *   **Администратор/HR:** Полный доступ к управлению пользователями, технологиями и квартальными циклами.
*   **F.5.1.4. Изоляция Данных:** Разработчики не должны иметь доступа к данным других сотрудников, за исключением агрегированных показателей на дашборде `Overview`. Тимлиды не видят детальную информацию по сотрудникам других команд.

#### **5.2. Управление Компетенциями и Оценками (Starmap Core)**
*   **F.5.2.1. Редактирование Самооценки:** Разработчики могут вносить и изменять свои оценки (`SkillRating` от 0 до 3) для каждой технологии в рамках текущего квартального цикла.
*   **F.5.2.2. Утверждение Оценок Тимлидом:** Тимлид может просматривать и корректировать самооценки своей команды. После утверждения оценки блокируются до следующего квартального цикла.
*   **F.5.2.3. История Оценок:** Система должна хранить историю всех оценок по кварталам для каждого сотрудника и технологии.
*   **F.5.2.4. Управление Технологиями:** Администратор может добавлять, редактировать или удалять технологии, а также определять их критичность (высокая/обычная/низкая).

#### **5.3. Квартальный Цикл Starmap**
*   **F.5.3.1. Инициализация Цикла:** В начале каждого квартала (по расписанию или вручную администратором) система инициирует новый цикл сбора данных.
*   **F.5.3.2. Копирование Оценок:** Новый цикл должен начинаться с копирования предыдущих оценок сотрудника, чтобы он мог их скорректировать, а не заполнять заново.
*   **F.5.3.3. Блокировка Оценок:** После завершения цикла (например, через 2 недели) все оценки блокируются для редактирования и становятся частью исторического Snapshot.
*   **F.5.3.4. Уведомления:** Система должна отправлять уведомления (почта, Slack) о начале/завершении цикла, о необходимости заполнить/утвердить оценки.

#### **5.4. Дашборды и Аналитика**
*   **F.5.4.1. Дашборд `Overview` (Командный Уровень):**
    *   **Сводная Карта Рисков:** Таблица, отображающая технологии, их критичность, текущее количество экспертов, целевое количество экспертов, статус `Bus-factor risk` (с цветовой индикацией), `Trend` изменения за квартал, и `План действий`.
    *   **Метрики:** Отображение агрегированных показателей с динамикой за квартал: `Coverage Index`, `Maturity Index`, `Red Zones`.
    *   **Селектор Кварталов:** Возможность выбора отображаемого квартала.
*   **F.5.4.2. Дашборд `Team` (Для Тимлидов):**
    *   **Уровень Компетенций:** Сводная таблица оценок всех сотрудников команды по всем технологиям.
    *   **Динамика Компетенций:** Таблица, показывающая общий рост каждого сотрудника за выбранный квартал.
    *   **Индекс Универсальности:** Таблица с количеством технологий, освоенных каждым сотрудником на уровне >= 2.
    *   **Key Person Risk:** Таблица с количеством технологий, где каждый сотрудник является единственным экспертом.
    *   Графики: Радар-чарты (или аналоги) для визуализации профиля компетенций сотрудника.
*   **F.5.4.3. Персональный Дашборд (Для Разработчиков):**
    *   Отображает только собственные оценки, динамику роста и личные `Action Plans`.
    *   Должен включать графики, аналогичные `Team` Dashboard, но сфокусированные на данных одного пользователя.
    *   Индекс Критической Экспертизы для данного сотрудника.
*   **F.5.4.4. Фоновые Расчеты:** Все метрики и агрегированные данные должны пересчитываться фоновыми задачами (Solid Queue) после изменения оценок или завершения квартального цикла, чтобы дашборды всегда отображали актуальные данные.

#### **5.5. Планы Действий (Action Plans)**
*   **F.5.5.1. Создание Плана:** Тимлид может создавать `Action Plans` для сотрудников или технологий, описывающие конкретные шаги по развитию.
*   **F.5.5.2. Привязка к Целям:** `Action Plans` должны быть привязаны к конкретной технологии, сотруднику и/или кварталу.
*   **F.5.5.3. Отслеживание Прогресса:** Возможность помечать пункты плана как "в работе", "выполнено", "отложено".

### **6. Нефункциональные Требования**

*   **NF.6.1. Безопасность:**
    *   **LDAP/SSO:** Обязательная аутентификация через LDAP.
    *   **Pundit/Action Policy:** Строгое соблюдение заданной ролевой модели.
    *   **Log/Audit:** Отслеживание всех критических изменений (оценки, планы) с помощью `Audited` или аналогичного решения.
*   **NF.6.2. Производительность:**
    *   Все расчеты метрик и агрегатов выполняются асинхронно в фоне с использованием Solid Queue.
    *   Интерфейс должен быть отзывчивым, без длительных задержек при навигации и загрузке данных.
    *   Эффективное кэширование данных UI средствами Rails/Hotwire.
*   **NF.6.3. Надежность и Расширяемость:**
    *   **Версионирование:** Поддержка версионирования Starmap данных по кварталам (`QuarterSnapshot`).
    *   **Резервное копирование:** PostgreSQL, стандартные процедуры резервного копирования на уровне инфраструктуры.
    *   **Масштабируемость:** Архитектура должна позволять масштабировать количество команд и пользователей.
*   **NF.6.4. Удобство Пользователя (UX):**
    *   Интуитивно понятный и чистый пользовательский интерфейс.
    *   Адаптивность для работы на различных устройствах (браузеры).
    *   Четкое визуальное различие между editable и read-only областями.
*   **NF.6.5. Интеграция:**
    *   **Расширяемость LDAP:** Возможность дальнейшего расширения синхронизации для получения дополнительных атрибутов пользователя (например, должность).
    *   **Уведомления:** Настраиваемая интеграция с Mattermost для отправки уведомлений.

### **7. Определения и Глоссарий (Из Методологии Starmap)**

*   **Шкала компетенций (0-3):** строго придерживаться определений:
    *   **0 - Не имею представления:** "Слышал об этом, но на практике не сталкивался. Нужен онбординг с нуля".
    *   **1 - Имею представление:** "Могу выполнять простые задачи под присмотром или в паре с коллегой".
    *   **2 - Свободно владею:** "Могу самостоятельно взять задачу средней сложности и довести ее до production".
    *   **3 - Могу учить других:** "Могу объяснить архитектурные решения, провести код-ревью и быть ментором".
*   **Критичность технологий:** (высокая/обычная/низкая) — определяется ответами на вопросы: влияние на бизнес, частота изменений, концентрация знаний.

### **8. Процесс Разработки**

*   Разработка ведется по Scrum/Kanban методологии.
*   Приоритеты определяются Unit Lead.
*   Обязательное покрытие тестами (unit, integration, feature).
